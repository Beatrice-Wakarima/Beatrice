01 Project Overview

Retail transactional data lived in disconnected, raw CSV sources with a 17.6% customer telemetry gap and silent lookup mapping errors. This project builds a fully containerized ELT pipeline — orchestrated by Apache Airflow, validated by pytest — that turns that raw data into a Dockerized, dimensional PostgreSQL warehouse feeding 7 analytical mart views and a Power BI reporting layer.

04 Business Problem

Retail transactional data lived in disconnected, raw CSV sources with severe structural anomalies — including a 17.6% customer telemetry gap, product description drift, and silent lookup table mapping errors. Without an automated framework, these issues caused broken table joins and corrupt revenue metrics in downstream dashboards.

05 Objectives

  • Resolve the 17.6% customer telemetry gap without discarding usable transaction rows.
  • Automate ELT with idempotent, retry-safe daily runs.
  • Catch data-quality failures automatically, before they reach a dashboard.
  • Package the entire stack for single-command, reproducible deployment.

06 Solution Overview

A fully containerized Staging → Star Schema → Mart Layer architecture runs under Docker Compose. Apache Airflow orchestrates idempotent Extract, Transform, Validate, and Load steps using a LocalExecutor, while a dedicated pytest suite and runtime SQL validation gates verify data integrity before the 7 mart views are ever queried by the Flask/Chart.js dashboard or Power BI template.

07 Engineering Decisions

  • Why Docker? Guarantees the entire stack — Airflow, PostgreSQL, pipeline code — runs identically anywhere with one command.
  • Why separate PostgreSQL instances? Isolates Airflow's internal metadata from the retail_dw business warehouse, so orchestration state never risks corrupting business data.
  • Why pytest over manual QA? Automated, repeatable validation catches regressions on every run, not just when someone remembers to check.
  • Why Apache Airflow's LocalExecutor? Right-sized for a single-host deployment without the operational overhead of distributed executors.
  • Why fail-fast validation gates? Stopping the pipeline on a threshold breach is cheaper than a business team acting on a wrong number.
  • Why both a Flask dashboard and a Power BI template? Flask/Chart.js for a lightweight public demo, Power BI for the format most business stakeholders already use.

11 Data Quality Strategy

A dedicated pytest suite and runtime SQL validation gates verify data integrity before dashboard ingestion. The daily pipeline halts execution automatically if critical business thresholds — like the customer telemetry gap — are breached, rather than silently propagating bad data downstream.

12 Business Impact

The entire BI infrastructure spins up with a single command, running a daily pipeline that halts execution if critical business thresholds fail. Downstream Power BI dashboards process cleanly structured, pre-aggregated semantic views instantly, eliminating dashboard drift and ensuring flawless financial sign preservation for retail returns.

14 Lessons Learned

  • Flagging missing data explicitly beats silently dropping rows — the gap itself became a trackable metric.
  • Separating orchestration metadata from business data prevents an entire class of accidental corruption.
  • Fail-fast validation is far cheaper to build than to explain after a bad number reaches leadership.
  • Docker Compose removed nearly all "works on my machine" friction during development.

15 Future Improvements

  • Add CI/CD via GitHub Actions to run the pytest suite on every commit.
  • Introduce incremental loading instead of full-table refreshes for larger datasets.
  • Add monitoring/alerting on Airflow DAG failures.
  • Migrate SQL transformations to dbt for versioned, testable models.
