HR & People Analytics Warehouse

A governed HR data warehouse that replaced four disconnected spreadsheet systems with one trusted reporting layer — built on Python, dbt, and BigQuery.

01 Project Overview

This project consolidates HR, payroll, and facilities records — previously scattered across four disconnected local systems in CSV, JSON, and Excel formats — into a single governed BigQuery warehouse. A Power Automate Desktop flow lands raw files on a schedule, dbt transforms them through a layered Staging → Intermediate → Marts architecture, and the resulting star schema feeds a Power BI reporting layer used for headcount, compensation, and workforce planning.

04 Business Problem

Critical human resource, payroll, and facilities data lived in isolated, mismatched formats across four disconnected local systems. The data suffered from severe structural anomalies: missing office foreign keys requiring manual inference, string-formatted salary data, missing header rows, and inconsistent naming conventions (e.g. "New-York" vs. "New York City"). Without an automated integration layer, manual reporting resulted in silent data drops, broken table joins, and conflicting employee metrics across offices.

05 Objectives

  • Automate ingestion of HR data from four disconnected local systems into a central warehouse.
  • Standardize inconsistent formats, naming conventions, and data types across sources.
  • Build a governed, testable star schema that supports reliable executive reporting.
  • Catch data-quality issues before they reach a dashboard, not after.

06 Solution Overview

A Power Automate Desktop flow monitors local files on a schedule and triggers a Python script to land raw datasets into BigQuery. From there, a modular dbt pipeline moves data through Staging, Intermediate, and Marts layers — standardizing types, resolving fuzzy-matched office names, and unifying mismatched JSON structures into a single employee grain. DuckDB validation gates run before data ever reaches the Power BI layer.

07 Engineering Decisions

  • Why BigQuery? Serverless scaling and native dbt support made it a low-overhead warehouse for a solo-maintained project.
  • Why dbt? Version-controlled, testable SQL transformations beat ad-hoc scripts for long-term maintainability.
  • Why a layered (Staging → Intermediate → Marts) architecture? Isolates messy fuzzy-matching logic from the final semantic models, so downstream reports never touch raw inconsistencies directly.
  • Why DuckDB for validation? Fast, local, in-process checks without adding warehouse compute cost for QA-only queries.
  • Why Power Automate Desktop over a cloud trigger? Matched the reality of HR files living on local systems without existing cloud integration.
  • Why a star schema? Employee/office/compensation facts and dimensions map cleanly to how HR actually asks questions — by department, location, and time period.

11 Data Quality Strategy

Custom singular dbt tests alongside DuckDB validation gates enforce uniqueness and referential integrity constraints, and run end-to-end row balance audits to catch accidental data fan-out or silent drops before dashboard ingestion — 38 tests run on every build.

12 Business Impact

The entire pipeline can be triggered locally or scheduled to run end-to-end without manual intervention. Downstream Power BI dashboards consume governed, pre-aggregated semantic views, eliminating manual file assembly, fixing dashboard drift, and giving HR a single trusted source for workforce reporting.

14 Lessons Learned

  • Layered architectures (Staging → Intermediate → Marts) pay off fast once fuzzy-matching logic gets messy.
  • Validation gates catch problems that manual spot-checking reliably misses.
  • Designing for the employee grain early avoided painful re-modeling later.
  • Local-to-cloud automation needs to match how the business actually stores its files, not an idealized version of it.

15 Future Improvements

  • Move orchestration from Power Automate Desktop to Apache Airflow for better observability.
  • Add CI/CD via GitHub Actions to run dbt tests on every pull request.
  • Introduce data lineage documentation via dbt docs.
  • Deploy the warehouse and dashboard refresh to a cloud-hosted schedule.
