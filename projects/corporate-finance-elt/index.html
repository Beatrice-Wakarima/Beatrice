04 Business Problem

Corporate financial data, revenue targets, capital budgeting schedules, and employee operational records lived in disconnected silos. There was no repeatable, unified way to combine these multi-domain datasets into an analytics warehouse without manual cleanup, resulting in "dashboard drift" and breaking changes in downstream executive reports.

05 Objectives

  • Consolidate 4 disconnected Excel-based domains into one PostgreSQL warehouse.
  • Guarantee idempotent, restartable pipeline runs with zero duplicate rows.
  • Decouple BI reporting logic from raw tables via a dedicated view layer.
  • Deliver a 6-page executive Power BI dashboard with reliable daily refresh.

06 Solution Overview

An Apache Airflow DAG orchestrates the extraction of 4 Excel sources into PostgreSQL, followed by a 5-layer SQL transformation process (36 transformation blocks) that builds a star schema. A Python-based view registry maps each transformation layer 1-to-1 to 22 structured presentation views, which a Power BI (.pbit) template consumes directly — so BI logic never touches raw tables.

07 Engineering Decisions

  • Why PostgreSQL? Strong support for structured multi-domain joins and mature tooling for a self-hosted analytics warehouse.
  • Why Apache Airflow? Needed dependency-aware scheduling and automatic retries across a 5-layer transformation chain.
  • Why a decoupled view registry? Keeps BI logic swappable without touching the underlying star schema — reports read from views, never raw tables.
  • Why idempotent loads? A mid-run failure shouldn't require a manual cleanup before the next run — restarts must be safe by default.
  • Why a star schema over a flat table? Multi-domain data (finance, budgeting, operations) needed shared conformed dimensions to stay joinable.
  • Why 5 modeling layers? Each layer isolates one transformation concern — raw cleanup, conforming, star schema, aggregation, presentation.

11 Data Quality Strategy

Idempotent load logic guarantees the pipeline restarts cleanly from any mid-execution failure point with zero duplicate rows — validated by row-count reconciliation checks between staging and star-schema layers on every run.

12 Business Impact

The entire corporate BI pipeline restarts cleanly from any mid-execution failure point with zero duplicate rows. Downstream Power BI report models ingest optimized metrics from structured views instantly, ensuring sub-second visual rendering, zero dashboard drift, and robust maintainability for executive reporting.

14 Lessons Learned

  • Idempotency needs to be designed in from the start — retrofitting it onto a running pipeline is much harder.
  • A decoupled view layer makes BI tooling swappable without touching the warehouse.
  • Conformed dimensions are what actually make "multi-domain" data joinable.
  • Airflow's dependency graph made debugging failures far faster than a linear script would have.

15 Future Improvements

  • Containerize the full stack with Docker Compose for one-command deployment.
  • Add CI/CD via GitHub Actions to validate SQL transformations on every commit.
  • Introduce automated data-quality testing (pytest or dbt tests) alongside the SQL layer.
  • Add monitoring/alerting on Airflow DAG failures.
